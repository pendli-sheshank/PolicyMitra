# Architecture Decision Log — PolicyMitra

Companion to `PRD.md`, `agents.md`, `skills.md`, `memory.md`. Records concrete implementation decisions made while turning those specs into code, especially judgment calls where the spec left room for interpretation.

## Decisions

1. **Synthetic corpus source format**: hand-authored Markdown (`meta.yaml` + `policy_wording.md` per insurer) using `<!-- clause: ID -->` markers and pipe-tables, instead of real/synthetic PDFs. This is fully deterministic to parse and lets us demonstrate the "table row never splits mid-chunk" quality bar (skills.md) convincingly without wrestling with PDF table extraction for content we're authoring ourselves. `ingestion/parsers/pdf_parser.py` exists as an interface-compatible stub for when real insurer PDFs are added later.

2. *(Superseded by #15.)* **Embedding dimension fixed at 384**, matching the no-API-key local fallback embedder (`local_hash_embedder.py`, a deterministic hash-based embedding). Embeddings are stored as packed little-endian float32 BLOBs (384 × 4 = 1536 bytes) in `kb_embeddings.embedding`, and dense search is a brute-force cosine scan in Python (`retrieval/dense_index.py`) — at a few hundred chunks a full scan is sub-10ms and an ANN index or numpy would be pure overhead. Swapping to a real provider (e.g. Voyage, 1024-dim) later just requires a re-embed; the BLOB carries no declared dimension.

3. *(Superseded by #15.)* **BM25 side implemented as SQLite FTS5** (`kb_chunks_fts` virtual table with `bm25()` ranking) rather than a separate `rank_bm25` in-process index or OpenSearch. Chunks already live in SQLite; a second index would need to stay in sync on every re-ingest for no real benefit at this scale. The FTS table is external-content against `kb_chunks`, synced by triggers, and uses `porter unicode61` tokenization — porter stemming mirrors the stemming the previous Postgres `'english'` text-search config applied, which kept retrieval recall identical (94.7%) across the migration. Query terms are OR-joined after stopword removal, matching real BM25 score-on-any-term behaviour.

4. **Hybrid merge via Reciprocal Rank Fusion (RRF)**, not weighted score-sum — FTS5 `bm25()` and cosine similarity aren't on comparable scales, and RRF (rank-based) avoids an arbitrary normalization constant. (Being rank-based is also what made the Postgres→SQLite scoring swap safe.)

5. **Citation contract is our own `[clause_id]` bracket-tag convention**, not a direct binding to Anthropic's Citations API response shape. The Q&A prompt instructs the model to tag every factual sentence with `[clause_id]`; Guardrail parses these tags mechanically. This keeps Guardrail's verification logic independent of the exact grounding API used — upgradeable to native Citations API spans later without changing the verification contract.

6. **Guardrail's numeric verification (steps 1–4: extract, citation-presence check, citation-membership check, normalized-value comparison) requires no LLM call at all** — pure regex/string matching. Only the *repair* step (rephrasing a single failed sentence) needs a live LLM. This means the core safety property ("never let an unverified number through") holds even with `NullLLMClient`: no key means no repair attempt, straight to a hard-coded redaction — a stricter, safer degrade, not a weaker one.

7. **Recommendation Agent ranking is deterministic Python** (`_score_plan()`), not LLM-driven. The LLM only phrases the one-line rationale/trade-off text, constrained to reference numbers already extracted by code. Guardrail still re-verifies those numbers independently. This structurally enforces the "every number must trace to a chunk, never the model's prior knowledge" requirement rather than relying on prompting alone.

8. **Session Memory TTL enforced without a scheduler**: `api/deps.py::expire_sessions()` runs a lazy sweep (`DELETE ... WHERE expires_at < now`) on every session read. Timestamps are stored as UTC ISO-8601 TEXT written through a single code path (`db/connection.py::utc_now_iso()` + adapters), so lexicographic comparison is chronological comparison.

9. **Agent (Module 3) identity is a placeholder header** (`X-Agent-Id`), not a real auth system — explicitly out of scope for this build. Every Layer-4 query filters by this identity server-side so cross-agent data leakage is structurally impossible even without real auth, but the identity itself is not verified/authenticated.

10. **Frontend is a single-page Next.js app** (App Router) with a mode switch (Chat / Recommend / Compare / Agent Copilot) rather than separate routes — the four flows are simple enough that client-side tab state is sufficient, avoiding routing complexity the spec doesn't call for.

11. **LLM/embedding clients are pluggable**: `agents/llm_client.py::get_llm_client()` returns a real `AnthropicClient` when `ANTHROPIC_API_KEY` is set, else a `NullLLMClient` that raises a typed `LLMNotConfiguredError` (never crashes). The API layer converts this to a clean `503`. The eval harness and Router/Q&A agents have explicit no-key degrade paths (keyword-fallback routing, offline numeric-traceability checks) so the system is meaningfully testable before any key is added.

12. *(Superseded by #15.)* **OpenRouter added as a second LLM provider**, alongside Anthropic, so a user can point the whole system at any model in OpenRouter's catalog (OpenAI, Google, Meta, Mistral, Anthropic itself, and others) via one API key. Selection logic in `get_llm_client()`: an explicit `LLM_PROVIDER=anthropic|openrouter` env var wins; if unset, `ANTHROPIC_API_KEY` is preferred when both are set (keeps prior default behavior unchanged for existing users), else `OPENROUTER_API_KEY`, else `NullLLMClient`. `OpenRouterClient` talks to OpenRouter's OpenAI-compatible `/chat/completions` endpoint directly via `httpx` (no extra SDK dependency) and returns the same `LLMResponse` shape as `AnthropicClient`, so every agent (`RouterAgent`, `QAAgent`, `RecommendationAgent`, `DraftingAgent`, `GuardrailAgent`'s repair step) works unchanged regardless of which provider is active — the choice is entirely a `.env` concern, never an agent-code concern. An explicit `LLM_PROVIDER` with its key missing degrades to `NullLLMClient` rather than silently falling back to the other provider, since that would violate the principle of least surprise for a user who deliberately chose a provider.

13. *(Superseded by #15.)* **Embedded SQLite over PostgreSQL+pgvector** (backend simplification): the entire data layer moved from a Postgres 16 + pgvector server to a single embedded SQLite file (`data/policymitra.db`, `POLICYMITRA_DB` to override). Rationale: every real-world setup failure this project hit (Codespaces sudo/peer-auth, hostname mismatches between docker-compose and bare environments, generated DB passwords tripping secret scanners) came from running a database *server*; at this corpus scale Postgres bought nothing that FTS5 + a brute-force cosine scan doesn't. There are now zero credentials anywhere in the stack, `make api` auto-migrates a missing DB file, and the eval gate confirmed retrieval recall unchanged (94.7% recall@5). Concurrency: WAL mode + 5s busy timeout + per-request connections; ingest wraps each document in one explicit transaction. A future scale-up (many insurers, real traffic) would revisit this, in which case the schema and SQL port back to Postgres mechanically (the schema files keep the old kb/mem/agent/audit separation as table-name prefixes).

14. **API hardening for forwarded/public dev environments**: (a) CORS allow-list is built at startup — localhost:3000 plus the GitHub Codespaces forwarded frontend origin derived from `CODESPACE_NAME`/`GITHUB_CODESPACES_PORT_FORWARDING_DOMAIN` — never a wildcard; (b) per-IP sliding-window rate limiting (`api/rate_limit.py`, pure-ASGI, in-memory, `RATE_LIMIT_PER_MINUTE` default 30, health endpoint exempt) protects a publicly forwarded port; (c) LLM keys are expected via real environment variables (e.g. Codespaces secrets) with `.env` as an optional local convenience — `load_dotenv` never overrides real env vars.

15. **Supabase Postgres + pgvector, OpenAI embeddings, Gemini chat** (supersedes #2, #3, #12, #13; deliberate product decision, reversing #13's SQLite direction). The data layer is Postgres with pgvector — a hosted Supabase project by default, which removes the local-DB-server pain that motivated #13 while restoring real vector search: `kb.embeddings.embedding VECTOR(1536)` with an HNSW cosine index (HNSW over ivfflat because a tiny corpus can't train meaningful ivfflat lists), BM25-style keyword search via Postgres `tsvector`/`ts_rank_cd` with the OR-rewritten tsquery from the original implementation. `DATABASE_URL` is required (no embedded fallback); connect through Supabase's session pooler (the direct DB host is IPv6-only, and `prepare_threshold=None` keeps psycopg pooler-safe). Providers are replaced, not extended: chat is Google Gemini only (`GeminiClient`, REST via httpx, `GEMINI_API_KEY`/`GEMINI_MODEL`), embeddings are OpenAI `text-embedding-3-small` only (`OpenAIEmbedder`, REST via httpx, `OPENAI_API_KEY`), with `ingestion.embedding.get_default_embedder()` as the single selection point shared by API, ingestion CLI, and eval harness so query-time and ingest-time vector spaces always match (dense search additionally filters on `embedder_name`). The no-key degrade contract from #11 is unchanged: `NullLLMClient` → typed 503, and the offline 1536-dim hash embedder (`local_hash_v2`) keeps retrieval, integration tests (`TEST_DATABASE_URL`, schema-reset per run), and the eval gate runnable without any LLM/embedding key. Session-memory timestamps returned to `TIMESTAMPTZ`/`now()` semantics (amending the ISO-TEXT mechanism in #8; the lazy-sweep design itself stands, via `mem.expire_sessions()`).
